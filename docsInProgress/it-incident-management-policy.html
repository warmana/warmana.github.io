Good incident management enables the MOJ to respond to security incidents in a calm, efficient and effective manner. This incident management policy describes what must be done when an incident occurs.

The Incident Management Standard [link to follow] provides help with creating an incident management plan that complies with the policy. For example, the standard describes the roles and responsibilities of people working to address the incident.

The policy statements are as follows:
POL.IMP.001 Each MoJ Business Group must have an IT Security Incident Management Plan which aligns to this policy. This plan must be common to all IT systems within a particular business group.
POL.IMP.002 All IT Security incidents or suspected incidents must be reported to the MoJ Operational Security Team (OST) within 60 minutes of detection.
POL.IMP.003 For all incidents involving an IT Security incident, an IT Security Incident Report Form must be completed and submitted to the OST [Ref, 10]. This is irrespective of the reporting route (i.e. a User direct with OST or a user via the IT helpdesk).
POL.IMP.004 All IT Security incidents involving personal data (or other information assets) must be reported to MoJ Data Access and Compliance Unit [Ref, 9].
POL.IMP.005 All IT Security incidents must be categorised in accordance with this policy.
POL.IMP.006 All MoJ staff must report any concerns that the MoJ ICT Security Policy [Ref, 1] is not being followed to their line manager.
POL.IMP.007 All MoJ staff must report any breach of the MoJ ICT Security Policy [Ref, 1] as an IT Security incident.
POL.IMP.008 All MoJ staff must report any suspicious activity which indicates an IT Security incident has occurred.
POL.IMP.009 All MoJ staff must report an IT Security incident either to the IT helpdesk or directly to the MoJ Operational Security Team using an ICT Security Incident Report Form.
POL.IMP.010 All MoJ Local Managers must ensure that all IT Security or personal data incidents or breaches are reported and taken seriously. These include facilitating any investigation and, where appropriate, pursue disciplinary action and/or legal proceedings.
POL.IMP.011 Each MoJ business group SIRO must ensure that each ICT domain (e.g. DISC or OMNI) which fall under their remit has an IT Security Incident Management Plan which implements this policy. A template plan and guidance is available in ICT Security – Incident Management Plan and Process guide [Ref, 11].
POL.IMP.012 All High impact IT Security incidents and any IT Security incident involving personal data must be reported to the SIRO immediately.
POL.IMP.013 All IT Security incidents involving the loss, theft or compromise of an Information Asset must be reported to the asset's IAO.
POL.IMP.014 Where the IT helpdesk receives a report of a security incident, this must be reported and escalated to the OST immediately.
POL.IMP.015 Each IT Security Incident Management Plan must include a pre-arranged escalation path where each stakeholder is named and aware of their role in the Incident Management Plan.
POL.ITSEC.016 The OST must maintain files on any investigation undertaken.
POL.ITSEC.017 Any diagnosis of an IT Security incident and the events surrounding it must be shared and reported to relevant stakeholders.
POL.ITSEC.018 An IT system which has a significant compromise (Medium or High impact, see POL.IMP.005) must be reported to the system Accreditor and a review of that system’s risk assessment and accreditation must be conducted.
POL.ITSEC.019 All IT Security incidents for an IT system must be collated and provided to the system Accreditor during the re-accreditation process.
POL.ITSEC.020 The IT Security Incident Management Plan for an IT System or overarching ICT Domain must include details on how that system or ICT domain IT services are restored (or recovered) following an IT Security incident.
POL.ITSEC.021 For each Medium and High impact (see POL.IMP.005) IT Security incident, a management report must be prepared covering:
  • A description of the incident;
  • The outcome of the incident investigation;
  • Actions raised (or taken) with associated action owners;
  • Any recommendations made.


This policy is dated October 2017, and is an update of the IT Incident Management Policy, v1.0, May 2013.
